# PDF操作

会話数: 11 | パターン数: 7

## 1. コンテキスト把握のための事前探索

具体的な操作（書き換え、移動、生成）を行う前に、必ず対象PDFの内容や周辺のソースコード構造を読み取って現状を把握する。

### 手順

1. 対象ファイルのパスと実在の確認
2. PDF内部のテキストや画像の抽出による内容確認
3. 関連するソースコード（Rust, Python等）の処理ロジックの読み込み

出現頻度: 11回

## 2. 中間成果物の排除とデータフローの直結

画像抽出などの処理過程で生成される一時的な中間フォルダ（Pフォルダ等）への依存を廃止し、メタデータ（JSON等）から直接PDFを操作する構造へ改善する。

### 手順

1. 既存のフォルダスキャン処理の依存箇所を特定
2. 入力ソースをJSON等の構造化データに切り替え
3. 不要になった中間ファイル生成ステップの削除

出現頻度: 2回

## 3. PDFを「唯一の正解ソース」とするメタデータ埋め込み

PDFを単なる出力物ではなく、解析データやGT（Ground Truth）情報を保持する媒体として扱い、埋め込まれたデータを読み戻して再利用可能にする（Smart PDF）。

### 手順

1. 解析データ（JSON等）のシリアライズ
2. PDFのメタデータ領域やXObjectへのデータ埋め込み
3. PDFからの埋め込みデータの抽出・復元機能の実装

出現頻度: 2回

## 4. ライブラリ特性を考慮した厳密な画像配置制御

PDF生成ライブラリ（printpdf等）の内部ID割り当て順序と画像挿入順序の不一致（ObjectIdバグ）を回避し、意図したページ・位置に正確に描画する。

### 手順

1. 画像オブジェクトと描画座標の明示的なマッピング管理
2. ID昇順などのライブラリ依存のソートロジックの修正
3. ページごとのコンテキストに応じた画像挿入順序の保証

出現頻度: 1回

## 5. ファイルシステム監視によるPDF処理の自動化

notifyクレート等を用いて特定フォルダを監視し、PDFの作成・更新をトリガーにバックグラウンドで解析やレビューを自動実行する。

### 手順

1. 監視対象ディレクトリと再帰設定の定義
2. ファイル作成（Create）イベントのキャッチとフィルタリング
3. LLM連携等の解析ジョブのキューイングと実行

出現頻度: 1回

## 6. UX向上のためのCLIからGUI（Tauri）へのワークフロー委譲

CLIでの解析結果表示（メモ帳等）の限界を回避するため、解析が必要なPDFを引数としてGUIアプリ（Tauri等）を起動し、リッチなUIで操作を完結させる。

### 手順

1. 右クリックメニュー等からのコマンドライン引数対応
2. 画像パスや解析コンテキストのアプリ内への受け渡し
3. アプリ内での解析結果プレビューと承認フローの実装

出現頻度: 1回

## 7. 外部スクリプトのネイティブ機能（Rust）への統合

Python/Pillow等で行っていたPDF生成・画像加工処理を、保守性と配布の容易さのためにRustクレート（printpdf等）に移植し統合する。

### 手順

1. 既存のPythonスクリプトのレイアウト・ロジックの解析
2. RustでのPDF生成エンジン（printpdf, image等）への再実装
3. CLIツールへのサブコマンドとしての組み込み

出現頻度: 1回
