# 温度管理写真サイクル

## 概要

アスファルト舗装工事における温度管理写真は、**決まったサイクル**で撮影される。
このスキルは、写真をサイクル単位で認識・管理するためのルールを定義する。

## 温度管理の構成

### 台ごとの3温度（1サイクル）

1台の合材（ダンプ1台分）につき、以下の3温度を測定・撮影する：

```
┌─────────────────────────────────────────────────────────┐
│ 1サイクル = 1台分の合材                                  │
├─────────────────────────────────────────────────────────┤
│ 1. 到着温度         - ダンプが現場到着時の合材温度      │
│ 2. 敷均し温度       - フィニッシャーで敷き均した直後    │
│ 3. 初期締固め前温度 - ローラーで初期転圧する前の温度    │
│    (別名: 初期転圧温度)                                  │
└─────────────────────────────────────────────────────────┘
```

### 開放温度（1日/施工場所につき1回）

```
┌─────────────────────────────────────────────────────────┐
│ 開放温度 - 交通開放時の舗装表面温度                      │
├─────────────────────────────────────────────────────────┤
│ - 施工場所（測点）に紐づく                              │
│ - 1日の施工終了時に1回撮影                              │
│ - 台数サイクルとは別管理                                │
└─────────────────────────────────────────────────────────┘
```

## 撮影パターン

各温度につき、3枚撮影される（順序は撮影者により異なる場合あり）：

```
到着温度 (3枚):
  ├─ 全景（作業状況全体）
  ├─ ボードアップ（黒板クローズアップ）
  └─ 温度計アップ（温度計の数値が読める）

敷均し温度 (3枚):
  ├─ 全景
  ├─ ボードアップ
  └─ 温度計アップ

初期締固め前温度 (3枚):
  ├─ 全景
  ├─ ボードアップ
  └─ 温度計アップ
```

**1サイクル = 3温度 × 3枚 = 9枚/台**

```
開放温度 (3枚) ※1日/施工場所につき1回:
  ├─ 全景
  ├─ ボードアップ
  └─ 温度計アップ
```

## 1日の施工における写真構成

```
1台目: 到着(3枚)→敷均し(3枚)→初期締固め前(3枚) = 9枚
2台目: 到着(3枚)→敷均し(3枚)→初期締固め前(3枚) = 9枚
3台目: 到着(3枚)→敷均し(3枚)→初期締固め前(3枚) = 9枚
─────────────────────────────────────────────────
台ごとの合計: 27枚（3台 × 9枚）

＋ 開放温度(3枚) ※施工終了時に1回
─────────────────────────────────────────────────
1日の総合計: 30枚
```

## 黒板の記載パターン

温度管理用黒板には以下が記載される：

```
┌────────────────────────────────────┐
│ アスファルト合材温度管理           │
├────────────────────────────────────┤
│ 工種: 表層工                       │
│ 日付: 11/27                        │
│ 種別: 再生密粒度As(13mm)           │
├────────────────────────────────────┤
│        1台目   2台目   3台目       │
│ 到着    161.1℃  ---     ---        │
│ 敷均し  155.3℃  ---     ---        │
│ 初期締固め前 148.8℃ ---  ---       │
└────────────────────────────────────┘
```

## 備考欄のルール

### 正しい備考

```
到着温度 161.1℃
敷均し温度 155.3℃
初期締固め前温度 148.8℃
アスファルト混合物温度測定 到着温度 161.1℃
```

### 禁止される備考

```
❌ 温度管理工      ← 「〜工」は工種・種別・細別のみ
❌ 温度測定工
❌ 温度測定        ← 測定値がない
❌ アスファルト温度管理  ← 具体的な値がない
```

## サイクル検出ロジック（実装指針）

```typescript
interface TemperaturePhotoSet {
  temp: number;              // 測定温度
  fullView: PhotoRecord;     // 全景
  boardUp: PhotoRecord;      // ボードアップ
  thermometer: PhotoRecord;  // 温度計アップ
}

// 台ごとのサイクル（9枚）
interface TemperatureCycle {
  cycleId: string;           // "TEMP_D1127_T1" (日付_台数)
  truckNumber: number;       // 1, 2, 3...
  date: string;              // "11/27"
  station: string;           // 測点

  arrival: TemperaturePhotoSet;           // 到着温度 (3枚)
  spread: TemperaturePhotoSet;            // 敷均し温度 (3枚)
  initialCompaction: TemperaturePhotoSet; // 初期締固め前温度 (3枚)
}

// 開放温度（1日/施工場所につき1回、3枚）
interface ReleaseTemperature {
  releaseId: string;         // "RELEASE_D1127_S1" (日付_測点)
  date: string;
  station: string;
  photos: TemperaturePhotoSet; // 開放温度 (3枚)
}

// 1日の温度管理全体
interface DailyTemperatureRecord {
  date: string;
  station: string;
  cycles: TemperatureCycle[];     // 台ごとのサイクル
  release: ReleaseTemperature;    // 開放温度（1回）
}
```

## サイクル認識の手順

1. **黒板OCRで温度種別を検出**
   - "到着温度" / "敷均し温度" / "初期締固め前温度" / "開放温度" を検出

2. **撮影時間でグループ化**
   - 同じ台の3温度は通常10〜30分以内に撮影される

3. **台数を特定**
   - 黒板の「1台目」「2台目」表記を読み取る
   - または時間順で推定

4. **サイクルIDを付与**
   - 同じサイクルの写真に同一IDを付与
   - これにより「このセットは1台目の温度管理」と認識

## 温度値の読み取り（AI解析の注意点）

### ボードアップから読め（温度計アップは読みづらい）

温度値を機械的に抽出する際、**ボードアップ（黒板クローズアップ）が最も信頼できるソース**。

- **ボードアップ**: 黒板に手書きで「到着温度 160.2℃」等と明記されている。テキストOCRで確実に読める
- **温度計アップ**: 棒状温度計やデジタル温度計の実物写真。AIにとってアナログ目盛りの読み取りは難しく、誤読しやすい
- **全景**: 作業状況全体が写っており、黒板の文字は小さくて読みにくい

### measurementsフィールドの罠

photo-ai-rustの`measurements`フィールドは黒板テキストから最初に見つかった温度値を拾う。
しかし**敷均し温度の黒板には到着温度も併記されている**ため、到着温度の値を誤って拾うことがある。

```
黒板の実際の記載（敷均し温度の黒板）:
  到着温度 160.7℃     ← measurementsがこっちを拾ってしまう
  敷均し温度 153.7℃   ← 本当に欲しい値
```

### 正しい温度値の取得方法

1. **ボードアップ写真（hasBoard=True）のdetectedTextから、focusTargetに対応する温度を正規表現で抽出する**
   - focusTarget=敷均し温度測定 → detectedTextから`敷均し温度\s*(\d+\.?\d*)℃`を抽出
   - focusTarget=初期締固め前温度測定 → `初期[転締][圧固]め?前?温度\s*(\d+\.?\d*)℃`

2. **measurementsフィールドを鵜呑みにしない**（特に敷均し・初期締固め前）

3. 到着温度のmeasurementsは通常正しい（黒板に到着温度しか書いてないため）

## 現状の問題点

1. **バッチ分断**: 6枚バッチで処理すると、1サイクル(9枚)が2バッチに分断され、文脈が途切れる

2. **サイクル認識なし**: 「到着→敷均し→初期締固め前」の順序関係を理解していない

3. **台数管理なし**: 何台目かを追跡していない

4. **統一問題**: 個別写真として処理するため、AIが「温度測定」に統一しようとする

## 解決策の方向性

1. **サイクル検出フェーズを追加**
   - バッチ処理の前に、写真群からサイクルを検出
   - 同じサイクルの写真に `cycleId` を付与

2. **サイクル単位でのコンテキスト伝播**
   - バッチ境界を超えて、同一サイクルの情報を共有

3. **備考の自動構成**
   - 温度種別 + 実測値を備考に自動記載
   - 例: "到着温度 161.1℃"

## 実運用手順（photo-ai-rust での温度管理PDF作成）

### Step 1: photo-ai analyze で初回解析
```bash
cd C:/Users/yuuji/photo-ai-rust
cargo run -- analyze "<温度管理フォルダ>" -o result.json -m master/construction_hierarchy.csv -w 舗装工 -t 品質管理写真 -b 9
```
- OCR/taggerは良好に動作（温度値・台数を正しく検出）
- workType/varietyのマスタ照合は失敗しやすい（温度管理用検索パターンがないため）

### Step 2: result.json を手動修正（Python/jq）
修正対象:
- **workType**: "舗装工" に統一
- **variety**: "切削オーバーレイ工" or "舗装打換え工"（工事による）
- **subphase**: "表層工"
- **station**: 施工日（"2月9日" 等）
- **remarks**: 温度種別+値+台数（例: "到着温度 159.7℃ (1台目)"）
- **measurements**: 温度値（例: "到着温度 159.7℃"）

3枚1組のグルーピング（ファイル番号の連番で判定）:
```python
groups = {
    (338,340): ("舗装日外気温 1℃", "外気温 1℃"),
    (341,343): ("到着温度 159.7℃ (1台目)", "到着温度 159.7℃"),
    # ... 3枚ずつ
}
```

### Step 3: export でPDF/Excel生成
```bash
cargo run -- export result.json --format both -o <出力先> -p 3
```
