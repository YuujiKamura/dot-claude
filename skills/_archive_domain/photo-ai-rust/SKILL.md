# photo-ai-rust

## 概要
Rust製の工事写真整理CLIツール。photo-tagger（Gemini AI）で写真を分類・グループ分けし、工種マスタと照合してPDF/Excelの写真台帳を生成する。

## リポジトリ
- パス: `C:/Users/yuuji/photo-ai-rust`
- GitHub: https://github.com/YuujiKamura/photo-ai-rust

## 現在の解析パイプライン（2026-02〜）

```
写真 → scan → photo-tagger（Gemini、必須） → マスタ照合 → 正規化 → export(PDF/Excel)
                    ↓                              ↓
            photo-groups.json              construction_hierarchy.csv
           （インクリメンタル）              （フォルダ名→備考で照合）
```

### パイプラインの設計思想
- **photo-taggerが必須**。結果が空なら即エラー終了。AI fallbackは無い
- **photo-groups.jsonは永続的**。既存の正しい結果は保持。未記録の写真だけ処理
- **フォルダ名＝写真カテゴリ**。フォルダ名でマスタの備考列を検索し、工種階層を決定
- **マスタ照合はフォルダ単位**。全写真のdetected_textを集約→1回だけマスタ照合→全写真に同一階層を割当
- **2フェーズワークフロー**: (1) 大分類=AI自動（tagger）、(2) 詳細内容=人間が記入

### 混在フォルダ（複数工種が混在する場合）
パイプラインはフォルダ単位で1つの階層しか割り当てられないため、1フォルダに複数工種の写真が混在する場合（例: 路面切削工→表層工の施工推移）は以下の手順:
1. `analyze` でphoto-tagger実行（OCR・グループ化は正しく動く）
2. result.jsonの各写真のworkType/variety/subphase/remarksを手修正（黒板OCR結果を参照）
3. `export` でPDF生成

### 重要: やってはいけないこと
- photo-groups.jsonを削除して再実行 → 正しい結果が消える
- AI解析（Claude CLI）にフォールバック → 廃止済み
- フォルダ名以外のソースでマスタ照合 → パイプライン複雑化の原因

## ★ analyze実行前の必須チェック

**analyzeを叩く前に、必ずマスタCSVを確認しろ。マスタに無い活動はゴミマッチにしかならない。**

```bash
# 1. フォルダ名でマスタを検索
grep "区画線保管" master/by_work_type/区画線工.csv
# → 該当なし = このまま実行しても的外れな結果になる

# 2. 該当なしなら行を追加してからanalyze
# フォルダ名の内容に合った備考・写真区分・検索パターンを設定する
# 例: 材料保管写真なら写真区分=使用材料写真、備考=材料保管状況

# 3. 追加後にanalyze実行
cargo run -- analyze <folder> --master master/construction_hierarchy.csv
```

チェックを怠るとどうなるか:
- 「区画線保管」（材料保管写真）を実行 → マスタに無い → 「プライマー散布状況」に誤マッチ
- 修正にかかる時間 > 事前にマスタ1行追加する時間

pub fn run_grouping(folder: &Path, batch_size: usize) -> Result<GroupRecords> {
    let mut records = load_group_records(folder);  // 既存のphoto-groups.json読み込み

    // ... pending のみ Gemini API で分類 ...

### 切削オーバーレイ工 vs 舗装打換え工の違い
- **切削オーバーレイ工**: 既設舗装を切削して上からオーバーレイ。タックコート乳剤散布（養生砂なし）
- **舗装打換え工**: 既設舗装を撤去して新設。プライムコート乳剤散布＋養生砂散布＋清掃

### 種別の統一ルール（小峯工事）
設計書の積算体系では「舗装打換え工」だが、**実態は切削オーバーレイ工事**。
種別（variety）は全て **切削オーバーレイ工** に統一する。
- 温度管理・コアー・散布量試験 → 切削オーバーレイ工（舗装打換え工にしない）
- 切削出来形 → 路面切削工（これは正しい。切削工程の写真）
- 施工状況 → 写真ごとに異なる（切削オーバーレイ工/路面切削工/区画線工が混在。これも正しい）
- マスタCSVで舗装打換え工と切削オーバーレイ工の検索パターンが同一のため、デフォルトが舗装打換え工に倒れやすい → パイプライン側で補正が必要

### マスタ照合の仕組み
- フォルダ名→マスタの備考列で完全一致検索
- 一致しない場合→検索パターン列で部分一致検索
- **新しいフォルダ種別が出たらマスタCSVに行を追加してからanalyzeを実行する**
- result.jsonを手書きせず、必ずパイプライン経由で生成すること

### マスタ照合が間違い/空欄になる原因（調査の優先順位）
remarksが間違っている or 空欄のresult.jsonを見たら、**簡単な原因から順に調べろ**:

1. **最初に検索パターンの有無を確認**: マスタCSVにその活動の行・検索パターンが存在するか？ `grep` で一発。無ければ行を追加するだけで終わり
   - 典型例: 「安全訓練」がマスタに無い → bigramで「安全パトロール」に誤マッチ
   - 典型例: 「切削完了」「端部舗装版撤去状況」がマスタに無い → 空欄
   - **対処: マスタCSVに行+検索パターンを追加してre-analyze**
2. **次にOCRパース結果を確認**: detectedTextの「工種:」にゴミが入っていないか
   - 2026-02修正済: カンマ残り対策（work_type_hint=Some(",")で全行除外されるバグ）
3. コードのバグ調査は最後。**まずマスタを見ろ**

**重要: コードを読む前にマスタCSVをgrepしろ。9割はマスタの行追加で解決する。**

### 測点（station）の表記変換ルール
黒板OCRの略記を正式名称に変換する:
- **L → 左車線**（例: `No.4 L` → `No.4 左車線`）
- **R → 右車線**（例: `No.0 R` → `No.0 右車線`）
- 仮区画線の場合は測点にマーク種別を書く（例: `ダイヤマーク`、`横断歩道`）

## 夜間工事のstation日付ズレ問題

### 問題
夜間工事では日付を跨いで撮影するため、EXIF日付が実際の作業日の翌日になる:
- フォルダ `0209切削` = 2月9日の作業 → EXIF日付は `2026-02-10`（深夜0時〜4時撮影）
- フォルダ `0212切削` = 2月12日の作業 → EXIF日付は `2026-02-13`

### 作業日の判断基準（優先順位）
1. **黒板のdetectedText** — 最も信頼できる。`令和8年2月9日` `舗設日 令和 8年 2月 12日` 等。現場で人間が書いた日付
2. **フォルダ名** — `0209切削` → 2月9日。通常は黒板と一致する
3. **EXIF日付** — **使うな**。夜間工事で必ずズレる

フォルダ名が無い or 不正な場合は、detectedTextから黒板の日付を読み取れ。
黒板とフォルダ名が矛盾する場合は黒板が正（フォルダ名のtypoの可能性）。

- **正**: `"station": "2月9日"` (黒板/フォルダ0209切削)
- **誤**: `"station": "2月10日"` (EXIF日付)
- パイプラインがEXIF日付ベースでstationを生成すると必ず間違う。result.jsonのstation修正が必要

### 温度管理写真のstation書式
```
外気温:         "2月9日"        （台数なし）
到着温度:       "2月9日 1台目"   （台数あり）
敷均し温度:     "2月9日 1台目"
初期締固め前温度: "2月9日 1台目"
開放温度:       "2月9日"        （台数なし）
```
- 台数はダンプトラックの通し番号（1台目, 6台目, 11台目 etc.）
- 外気温・開放温度は作業全体に対する測定なので台数なし
- 黒板の `ダンプ台数 N台目` から台数を読み取る

### exportのPDFファイル名もEXIF日付ベースになる
- `export` コマンドは `品質管理_温度管理_0210.pdf` のようにEXIF日付でファイル名を生成
- まとめフォルダにコピーする際、フォルダ名に合わせてリネームが必要
  - 例: `品質管理_温度管理_0210.pdf` → `品質管理_温度管理_0209.pdf`

### 修正手順
1. result.jsonのstation値を修正（日付＋台数）
2. `cargo run -- export result.json --format pdf` で再生成
3. まとめフォルダにフォルダ名ベースのファイル名でコピー

## 設計原則
- **PDF/Excelが最終成果物** - JSONは中間ファイル
- **photo-tagger必須** - AI fallbackなし、結果空ならエラー終了
- **photo-groups.jsonは消さない** - インクリメンタル、既存結果を保持
- **フォルダ名がカテゴリ** - フォルダ名→マスタ備考で照合
- **PRは作成しない** - masterブランチに直接プッシュ

## 俯瞰原則: コード変更時の影響範囲

### レンダリングコード変更 → 全アルバム再エクスポート
PDF/Excelの出力に影響するコード（pdf.rs, excel.rs, layout.rs, normalizer/等）を変更したら、**その工事で既にエクスポート済みの全アルバム**を出し直せ。目の前の1つだけ直して終わりにするな。

やるべきこと:
1. 変更したコードの影響範囲を考えろ（フォントサイズ変更→全PDF、出来形パーサー変更→出来形のみ、等）
2. 工事フォルダ配下の全result.jsonを探し、影響するものを全て再エクスポート
3. 1つ出し直して完了報告するな。「他にも影響するアルバムがないか」を自分で確認しろ

```bash
# 影響する全result.jsonを探す例
find "H:/マイドライブ/〇市道.../１５工事写真/" -name "result.json" -type f
```

### result.json変更 → そのアルバムだけ再エクスポート
result.jsonの内容（測定値、備考、測点等）を修正した場合は、そのresult.jsonのアルバムだけ再エクスポートすればよい。

### 一般原則: 木を見て森を見失うな
- 1つの修正に集中するあまり、同じ修正が必要な他のファイル/アルバムを忘れるな
- 「この変更は他のどこに影響するか？」を常に自問しろ
- ユーザーが「出し直して」と言ったら、言われたものだけでなく、同じ変更が必要な全体を出し直せ

## まとめPDFの命名規則

### ルール: `{写真区分}_{活動名}[_{日付MMDD}].pdf`
- 写真区分はresult.jsonの `photoCategory` フィールドから取る（"写真"を除去）
- マスタCSVの `写真区分` 列が元データ

### 例
| photoCategory | ファイル名 |
|---|---|
| 施工状況写真 | `施工状況_0209.pdf` |
| 品質管理写真 | `品質管理_温度管理_0211.pdf` |
| 出来形管理写真 | `出来形管理_コアー採取.pdf` |
| 安全管理写真 | `安全管理_安全パトロール_0211.pdf` |

### まとめフォルダへのコピー手順
1. **あてずっぽうで命名するな**。必ずソースを確認しろ
2. result.jsonの `photoCategory` を見る。無ければマスタCSVの `写真区分` 列を `grep` で引く
3. 日付はフォルダ名から取る。複数日にまたがるまとめは日付なし

### ★ 重複コピー禁止
同じ日付・同じ写真区分のPDFが複数ある場合、**活動名付きの具体的なPDFだけコピーしろ**。汎用名のPDFは中身が重複している。

```
# 0220フォルダに4つのPDFがある場合:
施工状況_道路付属施設工_0220.pdf  ← コピーする（具体的な活動名あり）
安全管理_交通保安施設_0220.pdf    ← コピーする（具体的な活動名あり）
施工状況_20260220_0220.pdf        ← コピーしない（汎用名＝上と中身が被る）
安全管理_20260220_0220.pdf        ← コピーしない（汎用名＝上と中身が被る）
```

判断基準: ファイル名に`YYYYMMDD`の日付だけで活動名がないPDFは、同日の具体名PDFと重複している。まとめに既に同じ写真区分＋日付のPDFがあれば不要。

## GT駆動TDD（Issue #97〜）

### 原則
- **GTが正解。パイプラインをGTに合わせる。GTをパイプラインに合わせるな**
- 場当たり的な後付処理（`if コアー then ...`）ではなく、パイプラインの設計レベルで解決する
- 1フィールド1テストケース。GTの各フィールドをアサーションとして書き、Red→Green→Refactorで進める

### TDDサイクル
1. GTからアサーションを書く（例: `assert_eq!(result.remarks, "コアー厚さ測定")`)
2. パイプライン通す → 失敗（Red）
3. パイプラインのロジックを修正 → 成功（Green）
4. 他テストが壊れてないか確認（Refactor）

### GTの場所
- `tests/ground_truth/accuracy_eval/*.json` — 手修正済み正解データ（PDFから抽出）
- 各JSONはフェーズ別/フォルダ別の写真群。フィールド: station, remarks, variety, measurements, workType, subphase, photoCategory, focusTarget

### 現状のギャップ（コアー写真で確認済み）
| フィールド | 根本原因 | 設計レベルの解法 |
|-----------|---------|----------------|
| measurements | パイプラインが出来形管理用紙を読まない | 出来形用紙OCR or 手入力フロー |
| station | 「取付道路」等の文脈をOCRから取れない、余計な車線付与 | コアー写真のstation正規化ルール |
| remarks | OCR結果をそのまま使用、マスタ名と不一致 | remarks正規化マッピング（舗装厚測定→コアー厚さ測定） |

## pair-completion（着手前/竣工ペアリング）

### 現状: フォルダ振り分け方式が実用的
自動ペアリングは精度不足。人間が手動振り分けするのが速くて正確。

```bash
# 1. PDFから測点テキスト抽出 → フォルダ作成（Pythonスクリプト）
# 2. ユーザーが竣工写真を該当フォルダにドラッグ&ドロップ
# 3. フォルダ構造からマッピングJSON生成
```

### AI画像マッチングの知見（Issue #101）
- **テキスト特徴抽出→照合は使えない**: 「背景に建物がある」のようなモヤッとした印象に収束。21枚の似た道路写真を区別できない
- **タグの創発 vs 選択**: AIにタグを自由記述させると解析精度がユニークにならず、どこかで見たような印象に寄る。選択肢を渡して選ばせるほうが精度が出る（Issue #94と同じ構造）
- **サムネイル300x200で21枚同時比較**: 10%正解。特定の1枚に回答が偏る
- **候補3枚なら正解する**: パース認識能力自体はある。問題は候補数

### パースライン特徴量（将来の改善方向）
人間がサムネイルで一瞬で判別できる理由は**パースラインの分布**:
- 消失点、建物稜線、窓格子、電柱の方向と本数
- 座標ベースなのでテキスト特徴と違い曖昧さがない
- ファジーな一致判定（「だいたい同じ方向に同じ本数」）で十分ユニーク
- 施工前後で路面は変わるがパースラインの源泉（建物、電柱）は不変

### AIの限界
- 個別のパース解析はできるが**統合推論ができない**
- 路面文字の頭が手前/奥かで撮影方向判定可能だが、AIは指示しないと自発しない
- 車を運転する人間には当たり前の常識（路面文字は進行方向に読める）がAIにはない
