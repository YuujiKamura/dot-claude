# 施工体制

会話数: 2 | パターン数: 4

## 1. Tauriアプリ機能拡張ワークフロー

Rustバックエンドとフロントエンド（JS/HTML）の両面から新機能を統合的に追加・修正する基本パターン。

### 手順

1. プロジェクト全体のディレクトリ構造と既存のRustモジュール（lib.rs, main.rs）の依存関係を確認する
2. Cargo.tomlに必要なクレート（notify等）が定義されているか確認・追加する
3. Rust側でロジック（ファイル監視、CLI引数解析、外部API連携）を実装する
4. フロントエンド側（main.js, index.html）でUIおよびTauriイベント受信処理を実装する
5. cargo checkおよびnpm run tauri devを用いてバックエンド・フロントエンド間の整合性を検証する

出現頻度: 2回

## 2. 外部インターフェースのUI統合再設計

UX向上のため、バラバラだった外部ツール（CLI、メモ帳、スクリプト）の機能をTauriアプリのUI内に集約・統合するパターン。

### 手順

1. 現状の外部ツール連携における課題（文字化け、UXの分断）を特定する
2. 外部からのトリガー（右クリックメニュー、コマンドライン引数）をアプリ起動・イベント発行に変換する設計を行う
3. OS統合用の補助スクリプト（PowerShell等）を作成・修正する
4. 解析結果をアプリ内の専用ビューア（PDF埋め込み等）で表示する機能を実装する

出現頻度: 2回

## 3. ファイルシステム監視型自動解析パターン

特定のフォルダを監視し、新規ファイル作成や変更を検知してバックグラウンドでGemini等のAIレビューを実行するパターン。

### 手順

1. notifyクレートを使用してRecursiveMode::Recursiveでディレクトリ監視を開始する
2. EventKind::Create等のイベントから解析対象ファイルを特定する
3. 非同期スレッドで外部CLI（Gemini CLI等）を呼び出し、バックグラウンドで解析を実行する
4. 解析結果をTauriのウィンドウイベントとしてフロントエンドにプッシュ通知する

出現頻度: 1回

## 4. ビルド・デプロイ検証ルーチン

開発中の機能がリリースビルドでも正常に動作するか、OS統合機能を含めて確認する手順。

### 手順

1. cargo checkでRustの型安全性と依存関係をチェックする
2. npm run tauri build または cargo tauri build --release でインストーラー/実行ファイルを生成する
3. 生成された成果物に対し、OSのコンテキストメニュー登録（ps1）等の統合テストを行う

出現頻度: 2回
