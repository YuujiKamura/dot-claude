# 写真管理

会話数: 9 | パターン数: 5

## 1. マスタ連動型AI写真仕分け・階層化

工事種別マスタ（CSV）とAIを組み合わせ、現場写真を適切な工事名や工種（舗装工、切削工等）のフォルダ構造に自動分類するパターン。

### 手順

1. construction_hierarchy.csv や 舗装工.csv などのマスタデータをロードする
2. photo-ai や photo-tagger を用いてディレクトリ内の写真をAI解析する
3. AIがマスタに存在しない工種を検知した場合は新規提案を行う
4. 解析結果を photo-groups.json や result.json に書き出し、フォルダ分けを実行する

出現頻度: 5回

## 2. クラウドストレージ間連携パイプライン

Dropboxで撮影・収集した写真をAIで処理し、最終的な成果物（PDF/整理済み写真）をGoogle Driveの共有フォルダへ配置するワークフロー。

### 手順

1. dropbox-fetch 等の独自ツールでDropboxから未整理画像を取得する
2. Google Drive（H:/マイドライブ/）内の工事別フォルダをスキャンし、パスを特定する
3. AI解析・PDF生成後、成果物を「写真帳まとめ」フォルダ等に移動（cp/mv）する

出現頻度: 3回

## 3. マルチ言語間レイアウト設定のSSoT管理

GAS（TypeScript）とphoto-ai-rust（Rust）間で、写真帳のPDF/Excelレイアウト定数（余白、座標、単位換算）を一元管理する設計パターン。

### 手順

1. shared/layout-config/layout-config.json に共通定数とスキーマを定義する
2. generate-layout-ts.ts や generate-layout-rs.ts などのスクリプトを叩き、各言語用の定数ファイルを自動生成する
3. 各言語のビルド（npm run build / cargo build）を行い、型の不整合がないか検証する

出現頻度: 1回

## 4. CLIバックグラウンド処理とリソース監視

大量の画像解析をバックグラウンドプロセスとして実行し、メモリ使用量や完了状態を監視しながら処理を進める運用パターン。

### 手順

1. cargo build --release で最適化されたバイナリを作成する
2. photo-tagger.exe 等をバックグラウンド（&）で実行し、タイムアウト設定（2m等）を設ける
3. tasklist や ps でプロセスの生存を確認し、ヒープメモリ不足等でクラッシュした場合は再試行や分割処理を検討する

出現頻度: 2回

## 5. APIコスト削減型・実画像統合テスト

Gemini等の外部API呼び出しを最小限にしつつ、実画像を用いてCLIの解析品質と処理速度を一貫して検証するテスト手法。

### 手順

1. H:/マイドライブ内の特定フォルダ（温度管理等）から12枚程度の実画像をテストデータとして定義する
2. APIキーを使用しない（またはモック/キャッシュされた）CLIモードでテストを実行する
3. 出力されたPDF名やJSONの構造が期待値通りか、処理時間が回帰していないかを検証する

出現頻度: 2回
