# CLI & Tooling

Patterns: 1

## 1. COM Vtable Overhead vs. Detection-based MVP

Pragmatic decision criteria for extending Windows Terminal UI, prioritizing low-overhead detection over heavy XAML/COM bindings.

### Steps

1. Evaluate the 'COM Tax': If adding a dynamic menu to a XAML SplitButton requires 400+ lines of Vtable boilerplate and WinRT glue code, a detection-based MVP (detecting shell types via heuristics) is often preferable for maintainability.
2. Decouple Visuals from Logic: Define UI elements in XAML but defer complex menu generation to external agents or simplified detection paths to minimize the binary footprint of terminal extensions.

### Examples

```
<!-- TabViewRoot.xaml -->
<SplitButton x:Name="ShellSelectionButton" />
<!-- Logic handled via lightweight process detection instead of heavy DataBinding -->
```
